是的，除了 EC2 启动/refresh 报错之外，子网只剩 7 个可用 IP 还会带来一系列“连锁”隐患，最好尽快扩容或新建子网。主要影响如下：

1. **EC2 启动/重建/refresh 必然失败**  
   AWS 要求每个实例（含弹性网卡 ENI）在启动时就必须拿到至少 1 个 IP；库存 < 需求时，RunInstances 会直接返回  
   `InsufficientFreeAddressesInSubnet`/`Launch Failed` 等错误，这是你已经遇到的情况 。

2. **滚动更新、自动扩缩容（ASG）、弹性调度全部受阻**  
   蓝绿/滚动发布时，系统要先起新实例再下旧实例；剩余 7 个 IP 不足以撑住哪怕一次最小粒度的滚动，更新会卡死。  
   Auto Scaling、EKS/ECS 节点组、Spot 实例替换同样无法拿到 IP，导致扩容失败 。

3. **ELB、NAT Gateway、VPC Endpoint、RDS 只读副本等“附属”资源也起不来**  
   这些服务都会在子网里自动创建 ENI 并占用 IP：  
   - ALB/NLB 每可用区每子网 1 个 ENI  
   - NAT Gateway 每 AZ 1 个 ENI  
   - Interface VPC Endpoint 每 AZ 1 个 ENI  
   如果 IP 池见底，它们同样报 “Subnet has insufficient free addresses” 而创建失败 。

4. **容器集群（EKS/ECS）Pod/任务调度失败**  
   默认 VPC-CNI 会给每个 Pod 分配一个“真实”子网 IP；7 个余量连一个节点上的 Pod 都满足不了，集群会频繁出现 `Insufficient.AvailableIpAddress` 导致 Pod 一直 Pending 。  
   即使开启 Prefix Delegation，也需要至少一个空闲的 /28（16 IP）块才能前缀分配，7 个 IP 依旧不够。

5. **监控/日志/安全工具无法部署或漂移**  
   很多 DaemonSet/Agent（CloudWatch Agent、GuardDuty、Config、Inspector、第三方防火墙）都以 Pod/实例形式存在，IP 不足时它们同样无法调度，导致监控盲区。

6. **网络接口“碎片”导致可用 IP 进一步减少**  
   AWS 在子网内按 ENI 粒度分配，删除实例后对应 ENI 若未及时清理，会留下“碎片”IP；剩余可用 IP 越少，碎片越难合并，可用数量会加速下降。

7. **运维应急手段受限**  
   当线上实例故障需要“起新救旧”或做抓包调试时，无 IP 可用会拖长 MTTR；甚至无法启动跳板机排障。

### 建议的应急与根治措施  
1. 立即在该 AZ 新建一条更大 CIDR（/24 甚至 /22）的子网，把 ASG/EKS 节点组、ELB、NAT 等资源逐步迁过去；新旧子网同在一条路由表下即可平滑迁移，无需换 VPC。  
2. 对现有子网先做“IP 盘点”——在控制台 Subnet → ENI 列表里确认是否有僵尸 ENI，手动释放可快速回收部分地址。  
3. 容器场景优先开启 VPC-CNI 的 Prefix Delegation 或自定义网络模式，减少每个 Pod 对子网 IP 的直接消耗 。  
4. 使用 VPC IPAM + CloudWatch 告警，把可用 IP < 20 设为红线，提前自动通知或触发 Lambda 扩容 。  

一句话：**7 个可用 IP 在生产环境已属于“高危红线”**，不仅 EC2 refresh 会报错，任何需要新建 ENI 的 AWS 服务都可能罢工；尽快扩容子网才是根本解法。
